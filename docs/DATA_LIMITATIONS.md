# Data Limitations & Design Decisions

This document exists so nobody — client, mentor, or future me — has to
reverse-engineer what this dataset actually contains. Every claim below was
verified by re-running the pipeline against the shipped data, not copied
from an earlier (inflated) report.

## 1. Real coverage per ticker is uneven — this was the actual hard problem

The original assumption going into this project was "we need to handle a
known gap: data only available from 2002 onward." The real problem turned
out to be bigger than that. After removing a placeholder/synthetic
2002-monthly-bridge that an earlier iteration of this pipeline had silently
generated (see §3), here is what real, source-backed daily data actually
covers:

| Ticker | First real trading day | Last day | Trading days | Largest internal gap |
|---|---|---|---|---|
| ASII | 2002-01-01 | 2007-12-28 | 1,564 | 3 days (normal weekend/holiday) |
| PTBA | 2002-12-23 | 2007-12-28 | 1,310 | 3 days |
| UNVR | 2003-09-03 | 2007-12-28 | 1,128 | 3 days |
| BBCA | **2004-06-08** | 2007-12-28 | 929 | 3 days |
| TLKM | **2004-09-28** | 2007-12-28 | 849 | 3 days |

BBCA has no real daily data at all for 2002 or 2003, and only 149 trading
days in 2004 (vs. ~244 expected for a full year). TLKM is similar — zero
real coverage for 2002-2003. This means any cross-ticker analysis that
assumes uniform 2002-2007 daily coverage across all five names will
silently be wrong for BBCA and TLKM unless the consumer checks
`data/processed/ticker_availability.csv` first.

**Takeaway for anyone using this data:** check `ticker_availability.csv`
before doing anything that joins or compares across tickers by date.

## 2. Close prices are NOT re-adjusted for stock splits — and that's intentional

Four of the five tickers had documented stock splits between 2002-2007
(source: KSEI/IDX corporate action records, shipped as
`data/reference/corporate_action_split_log.csv`). The textbook approach is
to compute a `Close_Adj` column that divides historical pre-split prices by
the cumulative split ratio so the series is comparable over time.

This pipeline does **not** do that, because doing so was empirically wrong
for this data. Both ASII (1:10 split, 2004-03-01) and BBCA (1:2 split,
2007-06-25) have real price data spanning their split dates, and in both
cases the raw `Close` field trades smoothly through the split with no
discontinuity:

```
ASII around 2004-03-01:  555 -> 540 -> 535 -> 570 -> 570   (no ~90% drop expected from a 1:10 split)
BBCA around 2007-06-25:  535 -> 535 -> 535 -> 535 -> 535   (no ~50% drop expected from a 1:2 split)
```

This means the source feed's `Close` field is already split-clean — most
likely already split-adjusted upstream by the data provider. Applying a
second backward-adjustment on top of it (which an earlier version of this
pipeline did) doesn't fix anything; it manufactures a fake discontinuity
where none exists. I verified this by implementing the standard
backward-adjustment formula correctly and watching it turn a flat BBCA
price series into a fabricated 100% one-day jump.

**Decision:** ship raw `Close` as the single canonical price series.
`Close_Adj` exists in the schema as an explicit alias of `Close` for
interface compatibility with the data quality checker, not as an
independently computed value. The split log is shipped separately as
reference metadata (useful for share-count-based calculations) rather than
baked into a price transform.

UNVR and TLKM's splits (2003-08-06 and 2004-08-23) both fall inside the
coverage gap described in §1 — there's no real pre-split data left for
either ticker once the synthetic bridge is removed, so adjustment is moot
for them regardless.

## 3. The original 2002 "data gap fix" was synthetic data, and it's removed

An earlier version of this pipeline filled missing early-2002 months with
values generated by `np.random.seed(42)` random walks anchored to a
hand-picked starting price per ticker. This was a deliberate, tested
fallback mechanism in the original code — not a silent bug — but it was
never disclosed as synthetic in the documentation that described it, which
called it "complete" and "verified" data.

I removed all 48 rows sourced this way (12 months x 4 tickers) from the
shipped dataset. The real gap they were standing in for is now documented
honestly in §1 instead of papered over.

## 4. Dividend figures are shipped raw, unadjusted, for the same reason as §2

`data/processed/dividend_historis_2002_2007.csv` contains dividend amounts
as recorded, with no split-adjustment applied. An earlier
`dividend_adjusted.csv` divided pre-split dividends by the same cumulative
split factor used for the (incorrect) price adjustment in §2 — given that
adjustment approach was shown to be unreliable for prices, it isn't applied
to dividends either. If you need split-adjusted dividend yield, verify the
adjustment against an independent source before computing it — don't
assume the split log ratio is safe to apply blindly here, since the price
case shows the source data and the split log don't always agree about
what's already been adjusted.

## 5. Two known minor data quality flags, not investigated further

`pipeline/data_quality.py` flags 2 rows out of 5,780 (PTBA and TLKM, both
on 2007-02-02) where `Close` falls slightly outside the `[Low, High]` range
by a few points. This is disclosed rather than silently fixed because there
isn't a confirmed explanation yet (possible prior-close reference mismatch
at the source, possible rounding). It affects 0.03% of rows and doesn't
change any conclusion in this document.

## 6. Scope cut from this package

Margin-leverage backtesting, DRIP reinvestment simulation, Kelly-criterion
position sizing, and Sharpe ratio projections were part of earlier
exploration of this dataset but are **not included** in this package. That
work surfaced two unresolved bugs (a stock-split adjustment error, now
fixed and described in §2 above, and a dividend reinvestment calculation
that overstated returns) that were never fully run to ground. Rather than
present unverified simulation output, this package scopes itself to the
data engineering layer: acquisition, cleaning, validation, and honest
documentation of what the data does and doesn't support.
